Design Refactor 281-line Home page inline CSS into styles.css

The Home page injects a large tags$style(HTML(...)) block (lines 155-435) directly into the Rmd. Moving these rules into www/styles.css would centralize all styling, make future design changes easier, and reduce the Rmd's visual clutter. Component-specific class names are already well-scoped (e.g., .nav-card, .welcome-text) so the migration is low-risk.

Design Consolidate color tokens into CSS custom properties

The site uses overlapping but inconsistent color values across files. www/styles.css defines some tokens (--navy, --border), but the scorecards, Home page, and inline style blocks each hardcode their own hex values. A unified set of CSS variables (maroon, gold, navy, plus semantic tokens for accent, success, warning) defined once in styles.css and referenced everywhere would prevent palette drift and make future rebranding trivial.

Current examples of inconsistency: Home page maroon is #7B1A1A, scorecard maroon is #800000, category banner gradient starts at #2a0000. All are "maroon" but none match exactly.

Design Add subtle card hover effects and transitions

The Home page navigation cards, Fact Sheet cards, and Help cards are static rectangles. Adding a gentle transform: translateY(-2px) and box-shadow elevation on hover (with a 150ms ease transition) would provide visual feedback, signal interactivity, and bring the design closer to modern card-based interfaces. This is a small CSS change with high perceived-polish return.

Design Improve table row hover feedback

National Data and City Data tables have zebra striping but no hover highlight. A subtle tr:hover { background-color: var(--gold-light); } (a very light gold, around #faf5e6) would help users track their eye across wide rows, improve readability, and reinforce the gold accent palette.

Design Add smooth scroll behavior and scroll-margin for anchored sections

When navigating between tabs or jumping to sections, the transition is instantaneous. Adding html { scroll-behavior: smooth; } and scroll-margin-top on section headings would create a more polished feel, especially on the Home page and Help tab where users move between content blocks.

Design Enhance responsive typography with clamp()

Current font sizing uses fixed rem values with media query overrides at 640px. CSS clamp() (e.g., font-size: clamp(2rem, 4vw, 2.8rem) for h1) would provide fluid scaling across all viewport widths, eliminating the jarring size jump at breakpoints. This is especially beneficial for tablet users who fall between the current mobile and desktop sizes.

Accessibility Add skip-to-content link

Screen reader and keyboard users must tab through the entire navbar before reaching page content. A visually hidden "Skip to main content" link as the first focusable element (becoming visible on focus) is a WCAG 2.4.1 requirement and a quick win. The site already has role="main" landmarks to target.

Accessibility Add lang="en" to the rendered HTML document

Flexdashboard's Rmd-to-HTML pipeline does not automatically add lang="en" to the <html> element. This is a WCAG Level A requirement (Success Criterion 3.1.1). It can be addressed by adding a small JavaScript snippet in www/cssloaders.html that sets document.documentElement.lang = "en" on load.

www/cssloaders.html

Fix prompt
Add document.addEventListener('DOMContentLoaded', function() { document.documentElement.lang = 'en'; }); to the existing <script> block in www/cssloaders.html.
Accessibility Add visible keyboard navigation hints to interactive cards

Home page navigation cards and Fact Sheet cards use JavaScript click handlers. While keyboard users can activate them via Enter/Space (the underlying buttons have role="button" and tabindex), there is no visual affordance indicating keyboard operability. Adding a brief "Press Enter to navigate" aria-describedby or a subtle focus-state label would improve discoverability for keyboard-only users.

What's Already Strong

Infrastructure and design decisions that are working well and should be preserved.

  • ✓ Accessibility foundation -- ARIA labels on all interactive elements, role="main" landmarks, aria-live="polite" regions for dynamic content, and focus-visible outlines are already in place across all tabs.
  • ✓ Responsive design -- mobile breakpoints at 640px with stacking layouts, reduced padding, and appropriate font scaling for the Home page, scorecards, and navigation cards.
  • ✓ Print styles -- both the main stylesheet and scorecard CSS include @media print blocks with color-adjust, break-inside avoidance, and cleaned-up box shadows.
  • ✓ Reduced-motion respect -- the spinner animation in cssloaders.html includes a prefers-reduced-motion media query that halts animation for users who prefer it.
  • ✓ Data architecture -- clean separation of data loading (setup chunk), helper functions (scripts/functions.R), and rendering logic. Reactive expressions are well-documented with comments.
  • ✓ Scorecard infrastructure -- the Quarto-rendered state and category scorecards with parameterized rendering, download buttons, and error handling via try/catch is production-quality.
  • ✓ Canon font enforcement -- as of May 2, 2026, EB Garamond is loaded once via @import in www/styles.css and applied universally via a * selector with !important, covering all elements including dynamically injected Shiny widgets.
  • ✓ Map accessibility -- leaflet maps use role="img" with dynamic aria-label that updates based on selected variables, providing screen reader users with meaningful context.
